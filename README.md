# STM32F407 Generic Template

แม่แบบโปรเจกต์ (Project Template) สำหรับบอร์ดไมโครคอนโทรลเลอร์ตระกูล **STM32F407xxx** ที่เน้นความสะอาดของโค้ดและการดึงประสิทธิภาพของฮาร์ดแวร์ออกมาให้ได้มากที่สุด โดยไม่พึ่งพาระบบ Auto-generate ของ STM32CubeMX

โปรเจกต์นี้ถูกออกแบบมาเพื่อเป็นรากฐานที่แข็งแกร่งสำหรับการพัฒนาเฟิร์มแวร์ระดับมืออาชีพ โดยเฉพาะอย่างยิ่งงานที่เกี่ยวข้องกับระบบเสียง (Audio) การคำนวณสัญญาณ หรือการประมวลผลสัญญาณดิจิทัล (DSP)

## คุณสมบัติเด่น (Features)

- **STM32 LL API (Low-Layer API):** แทนที่ไลบรารี HAL แบบดั้งเดิมด้วยไลบรารี LL API อย่างเป็นทางการจาก STMicroelectronics เพื่อการควบคุมรีจิสเตอร์ (Register) ที่รวดเร็วและใช้หน่วยความจำน้อยที่สุด
- **ระบบบิลด์ CMake และ Ninja:** โครงสร้างการคอมไพล์ที่รวดเร็วแบบ Out-of-source build ไม่สร้างไฟล์ขยะปะปนกับซอร์สโค้ด และทำงานร่วมกับ Toolchain จาก STM32CubeCLT ได้อย่างสมบูรณ์บนระบบปฏิบัติการ Windows
- **ระบบปฏิบัติการเวลาจริง (RTOS):** ติดตั้งไลบรารี FreeRTOS-Kernel แบบ Submodule พร้อมใช้งานสำหรับการทำ Multitasking ทันที
- **ไลบรารีประมวลผลสัญญาณ (CMSIS-DSP):** รองรับคำสั่งทางคณิตศาสตร์ความเร็วสูงที่ดึงพลังของหน่วยประมวลผลจุดทศนิยม (FPU) แบบฮาร์ดแวร์ และชุดคำสั่ง SIMD บนสถาปัตยกรรม Cortex-M4F มาใช้ เช่น `arm_mult_f32`, `arm_sin_f32`, FIR, IIR และ FFT

## โครงสร้างโปรเจกต์ (Project Structure)

- `Drivers/` - เก็บไลบรารีพื้นฐานและ Submodule ที่จำเป็น
  - `CMSIS/` - ไลบรารีแกนหลักของสถาปัตยกรรม ARM Cortex
  - `STM32F4xx_HAL_Driver/` - ไลบรารีของ ST (ใช้งานเฉพาะส่วนที่เป็น LL API)
  - `CMSIS-DSP/` - ไลบรารีคณิตศาสตร์ประมวลผลสัญญาณจาก ARM
- `FreeRTOS-Kernel/` - ซอร์สโค้ดของระบบปฏิบัติการ FreeRTOS
- `Inc/` - โฟลเดอร์เก็บไฟล์ Header (`.h`) ของโปรเจกต์ (เช่น `FreeRTOSConfig.h`)
- `Src/` - โฟลเดอร์เก็บไฟล์ Source (`.c`, `.S`)
- `cmake/` - สคริปต์สำหรับการตั้งค่า Toolchain ของ CMake

## วิธีการบิลด์ (How to Build)

โปรเจกต์นี้ตั้งค่ามาเพื่อให้ทำงานได้อย่างราบรื่นกับ **VS Code** และ **STM32 VS Code Extension**

1. ตรวจสอบว่าในระบบมีการติดตั้ง **STM32CubeCLT** เรียบร้อยแล้ว
2. เปิดโฟลเดอร์โปรเจกต์นี้ด้วย VS Code
3. เปิด Command Palette (`Ctrl + Shift + P`):
   - พิมพ์และเลือกคำสั่ง `CMake: Configure` เพื่อตั้งค่าโปรเจกต์ในครั้งแรก
   - พิมพ์และเลือกคำสั่ง `CMake: Build` เพื่อทำการคอมไพล์
4. ไฟล์ไบนารีสำหรับนำไปแฟลชลงบอร์ด (`.elf`, `.bin`, `.hex`) จะถูกสร้างและจัดเก็บไว้ในโฟลเดอร์ `build/Debug/`

## ข้อมูลสภาพแวดล้อม (Environment Details)

- **ชิปเป้าหมาย:** STM32F407xxx (เช่น STM32F407VGT6 หรือเทียบเท่า)
- **ความถี่สัญญาณนาฬิการะบบ (System Clock):** 168 MHz (ตั้งค่าผ่านคริสตัลภายนอก HSE 8 MHz ด้วย LL API)
- **คอมไพเลอร์:** `arm-none-eabi-gcc`
- **ดีบั๊กเกอร์:** รองรับการทำงานคู่กับ ST-Link GDB Server

## การทำงานของ CMSIS-DSP ในตัวอย่างโครงการ

ในไฟล์ `main.c` มีการเพิ่มฟังก์ชัน `dsp_test_task` ซึ่งเป็น Task ของ FreeRTOS สำหรับทดสอบการทำงานของไลบรารี CMSIS-DSP โดยเฉพาะ โค้ดส่วนนี้แสดงให้เห็นถึงการดึงพลังของหน่วยประมวลผลจุดทศนิยม (FPU) ออกมาใช้งานจริง:

1. **การคูณอาร์เรย์ (Vector Multiplication):**
   ```c
   arm_mult_f32(a, b, result, 4);
   ```
   ฟังก์ชันนี้ใช้สำหรับคูณข้อมูลในอาร์เรย์ `a` และ `b` เข้าด้วยกันแล้วเก็บผลลัพธ์ลงใน `result` การทำงานภายใต้ไลบรารีนี้ประมวลผลผ่านชุดคำสั่ง **SIMD (Single Instruction, Multiple Data)** ของสถาปัตยกรรม Cortex-M4F ซึ่งสามารถประมวลผลข้อมูลหลายตัวพร้อมกันใน 1 สัญญาณนาฬิกา (Clock cycle) เหมาะสำหรับการประมวลผลเสียงระดับโปรแกรมมิ่ง

2. **การคำนวณตรีโกณมิติแบบรวดเร็ว (Fast Math):**
   ```c
   float32_t sin_val = arm_sin_f32(PI/4);
   ```
   ฟังก์ชันนี้ใช้ตารางค้นหา (Lookup Table) ร่วมกับการประมาณค่าในช่วง (Interpolation) ที่ปรับแต่งมาเพื่อฮาร์ดแวร์ ARM โดยเฉพาะ ทำให้สามารถคำนวณค่า Sine ได้เร็วกว่าการใช้ฟังก์ชัน `sin()` จากไลบรารีมาตรฐาน `<math.h>` ของภาษา C หลายเท่า

## ระบบควบคุมหน้าจอแสดงผล OLED (SH1106) แบบ Polling (Without DMA)

สำหรับสาขานี้ (**`feature/oled-without-dma`**) ได้รับการติดตั้งระบบแสดงผลหน้าจอ OLED ขนาด 128x64 พิกเซล (ชิปควบคุม SH1106) ผ่านบัส I2C1 ความเร็ว Fast Mode @ 400 kHz โดยทำงานแบบ CPU Polling (ไม่ใช้ DMA) มีคุณสมบัติดังนี้:

1. **บัฟเฟอร์เดี่ยวประหยัดแรม (Single Buffering):** 
   จองแรมขนาด 1 KB สำหรับบัฟเฟอร์แสดงผลเพียงก้อนเดียว ทำงานประมวลผลข้อมูลกราฟิกและอักษรทั้งหมดลงบนบัฟเฟอร์เดี่ยวก่อนเริ่มส่งข้อมูลออกทางหน้าจอ

2. **การโอนถ่ายข้อมูลผ่าน CPU Polling:**
   ส่งข้อมูลพิกเซลและคำสั่งควบคุมทั้งหมดไปยังตัวควบคุมหน้าจอด้วยการตรวจสอบบิตแฟล็กในลูป CPU แบบ Synchronous (เช่น การเช็คสถานะ SB, ADDR, TXE, BTF) ซึ่งมีความง่ายในการตรวจสอบการแก้ไขข้อผิดพลาดและโครงสร้างทำงานที่ไม่ซับซ้อน เหมาะสำหรับการศึกษาระดับล่างและการเชื่อมต่อพื้นฐาน

3. **ความเร็วบัส 400 kHz:**
   แม้ว่าจะใช้การควบคุมแบบ CPU Polling แต่บัสสัญญาณนาฬิกา I2C1 ได้ถูกอัปเกรดเป็นความเร็ว **400 kHz** (Fast Mode) เพื่อเพิ่มความเร็วในการโอนถ่ายข้อมูลและทำให้หน้าจอตอบสนองลื่นไหลได้ดียิ่งขึ้นเมื่อเทียบกับความเร็วมาตรฐาน 100 kHz
